# HTTP 상태코드
- - - - 



### HTTP 상태코드 


클라이언트가 보낸 요청의 처리 상태를 응답에서 알려주는 기능



##### 1XX : 요청이 수신되어 처리중

    - 거의 사용되지 않음


##### 2XX : 클라이언트의 요청을 성공적으로 처리



##### 3XX : 요청을 완료하기 위해 유저 에이전트의 추가 조치 필요

    - 3XX 응답의 결과에 Location 헤더가 있으면 Location 위치로 자동 이동(리다이렉트)
    

    
**리다이렉션 흐름**

경로가 /event   => /event2 로 변경 됐을 경우

사용자가 /event 로 접속할 경우

    1. /event 경로 요청

    2. /event는 더 이상 사용하지 않으므로 301 코드와 함께 Location에 바뀐 경로값을 담아 응답

    3. 클라이언트는 301코드를 받으면 Location에 있는 바뀐 경로로 다시 요청

    4. 서버는 200 응답 코드를 보냄



**리다이렉션 종류**

    - 영구 리다이렉션 : 특정 리소스의 URI가 영구적으로 이동
                    
      301 : 리다이렉트시 요청 메서드가 GET으로 변하고, 본문이 제거 될 수 있음

      308 : 301과 같은 기능이지만 리다이렉트시 요청 메서드와 본문 유지.
            실무에선 잘 사용하지 않음. 301 사용

        예를 들어 등록을 위해 POST를 사용했지만 301코드는 GET으로 바꾸기 때문에 POST에 담겨있던 BODY는 전부 사라짐
        (처음부터 다시 등록해야 함)



    - 일시 리다이렉션 : 일시적인 변경. ex) 주문 완료 후 주문 내역 화면으로 이동

        302 : 리다이렉트 요청 메서드가 GET으로 변하고 본문이 제거될 수 있음(default로 많이 사용)

        307 : 리다이렉트시 요청 메서드와 본문 유지(요청 메서드를 변경하면 안됨)

        303 : 리다이렉트시 요청 메서드가 GET으로 변경

        * 302 경우에는 본문이 제거 될 수 있지만 안될 수도 있음. GET으로 확실히 변해야 한다면 303 사용.

        * POST로 주문후에 새로고침을 하면 주문이 다시 들어가게 됨. 
          중복 주문이 발생하기 때문에 보통 서버에서 중복주문이 되지 않도록 막지만 클라이언트에서 한번더 막아주는 것이 좋음.

        * 307, 303를 권장하지만 대부분 302를 기본값으로 사용. GET으로 변해도 괜찮다면 302 사용해도 큰 문제는 없음



    - 특수 리다이렉션  
        
        300 : 거의 안 씀

        304 : 캐시를 목적으로 사용. 클라이언트에게 리소스가 수정되지 않았음을 알려줌. 
                클라이언튼느 로컬PC에 저장된 캐시를 재사용함.(캐시로 리다이렉트).
                304 응답은 응답 메시지 바디를 포함하면 안됨.(로컬캐시 사용해야 하므로)
                


##### 4XX : 클라이언트 오류

클라이언트가 잘못된 요청, 데이터를 보내서 오류가 생김

    401 : 인증되지 않음

    403 : 접근권한이 불충분하여 승인거부

    404 : 요청 리소스가 서버에 없음. 또는 클라이어튼가 권한이 부족한 리소스에 접근할때 해당 리소스 존재 여부 자체를 숨기고 싶을때 사용



##### 5XX : 서버오류

서버 문제로 오류 발생. 재시도 하면 성공 할 수도 있음(서버복구)

    503 : 서버가 일시적인 과부하 또는 예정된 작업으로 잠시 요청을 처리 할 수 없음
